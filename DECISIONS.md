## ADR-006: API 키는 `.env` + 비밀번호 관리자 이중 보관
- **날짜**: 2026-04-13
- **결정**: `.env`는 git에 올리지 않고, 별도로 1Password 등에 백업 보관
- **이유**: `.env`가 휘발되면 다른 PC에서 API 복원 불가. GitHub에 푸시되면 키 유출.
- **대안 기각**: git-crypt 같은 암호화 커밋 — 복잡성 대비 이득 낮음
- **영향**: 새 PC 셋업 시 비밀번호 관리자에서 키 4개 복사 필요
- **상태**: Active

## ADR-005: 세션 영속성을 위한 문서 4층 구조
- **날짜**: 2026-04-13
- **결정**: `CLAUDE.md` / `PLAN.md` / `DECISIONS.md` / `docs/api_reference.md` 4개 문서로 맥락 영속화
- **이유**: AI 세션은 휘발성 → 다른 PC에서 작업 재개 시 맥락 손실. 코드만으론 "왜"와 "함정"이 복원 안 됨.
- **대안 기각**: 
  - Claude memory 디렉토리에만 의존 — 로컬 전용이라 동기화 안 됨
  - README.md 하나에 몰아넣기 — 섞이면 가독성 하락
- **영향**: 세션 종료 시 `DECISIONS.md`에 결정 추가, `CLAUDE.md`에 함정 반영하는 루틴 정착 필요
- **상태**: Active

## ADR-004: 서울 매출 API 중 `VwsmAdstrdSelngW`(행정동)를 MVP 기본 API로 채택
- **날짜**: 2026-04-12
- **결정**: 4개의 서울 매출 API 중 행정동 단위를 MVP 주력으로 사용
- **후보 비교**:
  | API | 단위 | 건수 | 채택 여부 |
  |---|---|:-:|:-:|
  | `VwsmMegaSelngW` | 서울시 전체 | 1,764 | 전국 비교용 |
  | `VwsmSignguSelngW` | 자치구 | 43,043 | 보조 |
  | **`VwsmAdstrdSelngW`** | **행정동** | **16,718** | **★ 주력** |
  | `VwsmTrdarSelngW` | 상권 | — | **존재 안 함** |
- **이유**: 
  - 행정동 단위가 "상권"이라는 생활 감각과 가장 부합 (예: 서교동 = 홍대)
  - 국토교통부 실거래가, 소상공인진흥공단 API와 행정동 코드로 조인 가능
  - 자치구는 너무 거시적, 상권 단위 API는 부재
- **영향**: 행정동 코드(8자리) 기준으로 DB 설계. 상권 영역(`TbgisTrdarRelm`)은 시각화 보조로만 사용.
- **상태**: Active

## ADR-003: MVP 스코프 축소 — "임대료 분석"을 "매출 트렌드 분석"으로 피벗
- **날짜**: 2026-04-12
- **결정**: CONTEXT.md 원안의 "임대료 상승률 연 X%" 기능을 v1에서 제외. "매출 트렌드 기반 창업 추천"으로 대체.
- **이유**: 
  - 국토교통부 상업용 실거래가 API는 **매매가만** 제공, 상가 **전월세(임대료) 데이터 없음**
  - 등기부등본은 유료(대법원), 공공 API 미존재
  - 반면 서울 매출 API는 분기별 시계열 7년치(2019~2025) 확보 가능
- **대안 기각**: 
  - KB부동산 등 상업 데이터 구매 — 초기 비용 부담
  - 임대료 API 먼저 신청 후 대기 — MVP 시작 지연
- **영향**: 
  - v1 리포트 문구가 CONTEXT.md 예시와 달라짐 (임대료 언급 제거)
  - Phase 2에서 상업용 전월세 API 신청 후 임대료 분석 복원 예정
  - 대신 **매출 증감률, 객단가, 업종 트렌드, 시간대·성별·연령 분해**를 전면 배치
- **상태**: Active, Phase 2에서 보완 예정

## ADR-002: Sprint 0 접근법 — 기획 확정 전 API PoC 우선
- **날짜**: 2026-04-11
- **결정**: CONTEXT.md만 작성된 상태에서 기획 구체화 대신 공공데이터 API PoC를 먼저 실행
- **이유**: 
  - 공공 API 품질·커버리지는 호출해봐야 알 수 있음
  - 데이터가 없으면 기획이 바뀔 수밖에 없으니 검증이 우선
  - 1인 개발자에게 애자일(B 옵션)이 더 효율적
- **대안 기각**: 기획서·User Story를 먼저 확정하는 전통적 접근 — 데이터 가용성 리스크 선반영 안 됨
- **영향**: 
  - Sprint 0 결과가 ADR-003(MVP 피벗)의 근거가 됨
  - 검증 결과를 반영한 PLAN.md가 CONTEXT.md를 보완
- **상태**: Completed

## ADR-001: 기술 스택 — Python + FastAPI + PostgreSQL + Claude API
- **날짜**: 2026-04-11
- **결정**: 백엔드 Python(FastAPI), DB PostgreSQL, AI Claude API, 프론트 Next.js
- **이유**: 
  - 개발자가 Python/JS 경험 있음
  - FastAPI는 공공데이터 API 다수 호출·비동기 처리에 적합
  - PostgreSQL은 지리공간 데이터(PostGIS) 확장 가능
  - Claude API는 한국어 자연어 리포트 품질 우수
- **대안 기각**: 
  - Node.js 백엔드 — 데이터 분석·머신러닝 라이브러리 Python 대비 약함
  - SQLite — 분기별 수백만 건 누적되는 상권 데이터에 부적합
- **영향**: requirements.txt에 `fastapi`, `sqlalchemy`, `psycopg2`, `anthropic` 등 추가 예정
- **상태**: Active (v1 착수 시점에 적용)
